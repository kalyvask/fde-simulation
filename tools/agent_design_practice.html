Sierra Agent Design — 3-Lens Whiteboard Practice

5:00

CUSTOMER· 90s

  • Who
    specific persona, not "users"
    Loyalty program member, often returning, high-intent on trip planning, occasionally browsing.
  • Emotional state
    stressed / anxious / curious / overwhelmed / frustrated
    Curious (planning) OR anxious (close to expiration / status threshold). Different from a frustrated reservations customer.
  • JTBD — top 3 intents
    3 concrete jobs, prioritized
    (1) Check points balance. (2) Understand redemption options for an upcoming trip. (3) Understand status implications for booking decisions.
  • Why failing today
    specific friction, not generic UX
    Loyalty queries route to live agents → 30+ min wait time → customer either abandons or churns.
  • What's hard (root cause)
    structural issue, not symptom
    Customer doesn't know if their question is "simple" (lookup) or "nuanced" (which redemption is best). Has to assume the worst and call.

PRODUCT· 90s

  • Intent
    what the customer is trying to do
    Loyalty information + simple decision support.
  • In scope
    list 3-5 concrete capabilities
    Points balance, expiration dates, redemption options, tier benefits, status calculations, "what's my balance worth" comparisons.
  • Out of scope (equally explicit)
    define as explicitly as in-scope
    Account changes (no write), reservation transactions, M&A / company-financial speculation, anything requiring real-time external data, advisory recommendations ("which redemption is best for ME?").
  • Trust levels (act / ask / escalate)
    when does the agent act autonomously vs ask vs escalate
    Act: factual lookups (balance, dates, policy quotes). Ask: edge cases ("here are 3 redemption options — would you like me to compare them?"). Escalate: advisory queries, account-specific exceptions, anything ambiguous.
  • Fallback
    what happens when confidence is low
    "Let me connect you to a loyalty specialist" with one-click handoff; agent passes context; no repeating.
  • Metric + tension (name which you protect)
    containment vs CSAT, speed vs accuracy, breadth vs depth
    Primary: 70% containment within in-scope queries. TENSION: Containment vs CSAT — we protect CSAT (≥4.0 floor), would rather escalate borderline cases than risk a wrong answer that erodes trust.

TECHNICAL· 60s · keep light

  • Read access
    specific data sources
    Loyalty database (balance, tier, transaction history), policy documentation, FAQ corpus.
  • Write access
    usually NONE in v1
    NONE in v1. Only writes are to the escalation queue (logging human handoffs).
  • Data freshness
    live / cached / batched
    Balance read live; policy with 1-hour cache; FAQ indexed corpus.
  • ONE specific integration risk
    PII / latency / regulatory / data quality — pick one and detail it
    Stale policy documentation. If a loyalty policy changes (e.g., redemption rates), the agent could cite stale info. Mitigation: policy-version pinning + scheduled cache refresh + LLM-as-judge check before send.
  • Validation plan
    PoC + eval set + compliance + closed beta
    PoC + 50-case eval set (factual lookup + edge cases) + compliance sign-off + senior-analyst review of 20 sample drafts + closed beta with 100 customers.

Predictable Probes

Customer column

  • "Walk me through a difficult customer interaction. How does the agent handle frustration?"
  • "A customer was mis-handled by a previous AI rollout. How do you re-earn trust?"
  • "How do you avoid sounding like a chatbot?"

Product column

  • "What's the experience if the customer asks something out of scope?"
  • "If 30% of conversations are at the scope boundary, how do you decide what to add?"
  • "How do you measure containment quality, not just rate?"
  • "What does week-4 look like vs week-1?"

Technical column

  • "Why read-only? Defend it."
  • "Where's the human-in-the-loop in week 1 vs week 8?"
  • "What's the worst-case if data freshness is off by 24 hours?"
  • "How does compliance review work at scale?"

After you finish, ask yourself

  • Did you name the emotional state in Customer?
  • Did you define out-of-scope as explicitly as in-scope in Product?
  • Did you distinguish read vs write in Technical?
  • Did you name ONE specific risk, not a generic list?
  • Did you name the metric tension (and which you protect)?
  • Did you land a one-sentence synthesis at the end?

Quick Reference Rules

  • Name the emotional stateStressed / anxious / curious. This single move drives design decisions and signals product seniority.
  • Out-of-scope = in-scopeDefine both equally explicitly. Shows judgment, not laziness.
  • Name the metric tensionContainment vs CSAT. Speed vs accuracy. Pick which you protect in v1 and say why.
  • Distinguish read vs writeWrite almost always NONE in v1. Read-only v1 is the senior move.
  • One specific riskNot a generic list. "Documentation retrieval with policy-version pinning" beats "we'd use RAG and have guardrails."
  • Never jump columnsIf the interviewer probes out of sequence, park it: "Let me note that in the technical column and come back to it."
  • Pace yourself0:00–0:30 restate · 0:30–2:00 Customer · 2:00–3:30 Product · 3:30–4:30 Technical · 4:30–5:00 synthesize.
  • Then stopLet the interviewer probe. Their probes are the test.
